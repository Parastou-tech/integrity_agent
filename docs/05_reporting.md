# Reporting

Reports are generated by `report_generator.py` and persisted to the `reports` Cosmos DB container. Two types exist: SESSION and POST_LAB.

---

## SESSION Report

Generated automatically at `POST /session/end`, or manually via `POST /report/generate`.

**Function:** `generate_session_report(session_doc, cosmos)`

### Summary Fields

| Field | Type | Description |
|-------|------|-------------|
| `total_questions` | `int` | Total messages sent by the student in this session |
| `violation_count` | `int` | Number of DIRECT_SOLUTION or ANSWER_FARMING questions detected |
| `escalated` | `bool` | True if 3+ violations were reached |
| `final_status` | `FinalStatus` | CLEAN / WARNING / ESCALATED (see below) |
| `classification_distribution` | `dict` | Count of each `QuestionClassification` value |
| `concept_struggle_summary` | `list` | Concepts from violation questions with violation types |

### `final_status` Logic

```
escalated == true              → ESCALATED
escalated == false and violations > 0  → WARNING
no violations                  → CLEAN
```

### `classification_distribution` Example
```json
{
  "CONCEPTUAL": 6,
  "PROCEDURAL": 4,
  "CLARIFICATION": 1,
  "DIRECT_SOLUTION": 1,
  "ANSWER_FARMING": 0
}
```
Useful for visualising what kind of help-seeking behaviour the student showed.

### `concept_struggle_summary` Example
```json
[
  {
    "concept": "BJT biasing",
    "count": 2,
    "violation_types": ["DIRECT_SOLUTION_REQUEST"]
  },
  {
    "concept": "transfer function",
    "count": 1,
    "violation_types": ["DIRECT_SOLUTION_REQUEST", "ANSWER_FARMING"]
  }
]
```
Only concepts that appeared in **violation questions** are listed here. `violation_types` is a deduplicated list of which violation types that concept appeared in.

### Escalation Log
```json
{
  "escalated": true,
  "escalation_timestamp": "2026-01-15T14:45:03Z",
  "reason": "Violation threshold (3) exceeded. Final violation type: DIRECT_SOLUTION_REQUEST"
}
```
`escalation_timestamp` is taken from the third violation record's timestamp.

### Violations Detail
The full list of `ViolationRecord` dicts, including `question_text`, `violation_type`, `severity`, and `timestamp`.

### Instructor Notes
The `instructor_notes` field starts as an empty string `""`. Instructors write to it via `PATCH /report/{report_id}`. The field accepts any dict (structured notes) or string (free-form text).

### Raw Session Snapshot
The complete `SessionDocument` at the time the report was generated, included for audit purposes.

## POST_LAB Report

Generated on demand via `POST /report/post-lab`. Takes a list of session IDs for a single student's complete lab attempt (may span multiple sessions if the student closed and reopened).

**Function:** `generate_post_lab_report(student_id, session_docs, lab_id, cosmos)`

### Over-Reliance Indicators

Five boolean/structured indicators are computed across all sessions combined:

#### `high_direct_solution_ratio`
```
(count of DIRECT_SOLUTION classifications) / (total questions) > 0.2
```
Fires when more than 20% of a student's questions were attempts to get direct answers.

#### `rapid_successive_questions`
```
any two consecutive questions were < 30 seconds apart
```
Detected by sorting all questions by timestamp and checking inter-question gaps. Rapid bursts suggest the student is iterating to extract a solution rather than working through it themselves.

#### `escalated_any_session`
```
any session in the provided list has escalated == true
```
True if the student hit 3 violations in any single session.

#### `repeated_violation_types`
```
{ violation_type: count } for any violation type with count >= 3
```
Returns a dict of violation types that appeared 3+ times across all sessions. An empty dict means no type repeated enough to flag.

Example: `{ "DIRECT_SOLUTION_REQUEST": 4 }` — the student attempted to get direct solutions 4 times across their sessions.

#### `concept_struggle_areas`
```
[{ concept, count }] for concepts appearing in violation questions, sorted by count desc
```
Unlike the session-level `concept_struggle_summary`, this aggregates across all sessions. Concepts from violation questions only — shows which specific topics the student was trying to bypass.

Example:
```json
[
  { "concept": "BJT biasing", "count": 3 },
  { "concept": "transfer function", "count": 1 }
]
```

### `flagged_count` and `summary_text`

Each indicator that fires contributes 1 to `flagged_count`:
- 0 flags → `"No over-reliance indicators detected."`
- 1 flag → `"1 over-reliance indicator detected. Recommend instructor review."`
- 2+ flags → `"N over-reliance indicators detected. Strong recommendation for instructor review."`

### Stats
```json
{
  "total_questions": 28,
  "total_violations": 5,
  "sessions_analysed": 2
}
```

## Accessing Reports

| Action | Endpoint |
|--------|---------|
| Get a report by ID | `GET /report/{report_id}?student_id=<id>` |
| Add instructor notes | `PATCH /report/{report_id}` |
| Generate SESSION report manually | `POST /report/generate` |
| Generate POST_LAB report | `POST /report/post-lab` |

Reports are keyed by `report_id` (UUID) and partitioned by `student_id` in Cosmos. A session's `report_id` is returned in the `POST /session/end` response and stored on `SessionDocument.report_id`.
